# 테스트 해보기 단위 통합 부하 강의 01: 단위 테스트 설계

## 학습 배경
이 장은 **단위 테스트 설계**를 중심으로 진행하며, 다음을 목표로 합니다: 테스트 피라미드를 이해하고 품질을 수치로 관리.

## 이 장에서 답해야 할 질문
1. 단위 테스트 설계는 어떤 문제를 해결하기 위해 등장하는가?
2. 단위/통합/부하 테스트의 목적과 비용 구조 중 어떤 지점에서 단위 테스트 설계가 핵심 역할을 하는가?
3. 프론트 코드와 연결할 때 계약(API/이벤트/에러 포맷)을 어떻게 맞출 것인가?

## 교과서 핵심 개념
### A. 문제 상황 정의
실무에서는 기능 구현 자체보다, 실패했을 때 원인을 빠르게 찾고 복구하는 구조가 중요합니다. 단위 테스트 설계는 이 문제를 해결하기 위한 기본 축입니다.

### B. 동작 메커니즘
단위/통합/부하 테스트의 목적과 비용 구조 관점에서 보면, 단위 테스트 설계는 입력 검증, 실행 분기, 상태 전이, 응답 표준화 과정에 직접 관여합니다. 즉 "동작"뿐 아니라 "운영 안정성"까지 결정합니다.

### C. 설계 기준
- 경계 분리: 라우팅/비즈니스/저장소 책임 분리
- 실패 우선: 정상 흐름보다 실패 흐름을 먼저 설계
- 관측 가능성: 로그/메트릭/트레이스에 남는 구조

## 요청-처리-응답 흐름에서의 위치
```text
요구사항 정의 -> 테스트 작성 -> 실행/리포트 -> 회귀 방지
```

## 코드 워크스루
```js
// 주제: 단위 테스트 설계
async function main() {
  try {
    console.log('[start]', '단위 테스트 설계');
    // TODO: 실제 로직 연결
    console.log('[done]', '단위 테스트 설계');
  } catch (error) {
    console.error('[error]', error.message);
  }
}

main();
```

### 코드 해설
1. 진입점에서 입력과 실행환경을 명시적으로 고정합니다.
2. 핵심 처리 로직은 테스트 가능한 함수 단위로 분리합니다.
3. 예외 케이스를 공통 포맷으로 정리해 프론트에서 일관되게 처리할 수 있게 만듭니다.

## 실습 미션
1. 코드 실행 후 성공/실패 로그를 각각 1회 이상 수집한다.
2. 프론트(React/Next)에서 호출할 API 혹은 함수 인터페이스를 문서화한다.
3. 입력 검증 규칙 1개와 에러 응답 스키마 1개를 추가한다.

### 완료 기준
- [ ] 실패 케이스 재현 경로를 문서화했다.
- [ ] 응답 형식(성공/실패)이 일관된다.
- [ ] 다음 리팩터링 TODO 3개를 작성했다.

## 프론트엔드 개발자 관점의 연결 포인트
- 컴포넌트 상태 설계 전, 서버 응답 스키마(성공/실패)를 먼저 확정한다.
- 낙관적 UI를 쓰는 경우, 롤백 시나리오를 서버 에러코드와 함께 설계한다.
- 네트워크 재시도 정책과 서버 idempotency(중복처리 방지)를 함께 맞춘다.

## 실무에서 자주 하는 실수
- E2E만 의존하고 단위 테스트를 생략
- 외부 의존(Mock) 경계 없이 flaky 테스트 생성
- 부하 테스트 없이 운영 투입
