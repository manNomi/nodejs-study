# 테스트 해보기 단위 통합 부하 강의 03. Mock/Stub 전략

## 학습 개요
Mock/Stub 전략는 테스트를 기능 검증 도구가 아니라 변경 안정장치로 다루는 방법을 설명한다.

## 왜 중요한가
배포가 빨라질수록 회귀 버그 위험도 커진다. 테스트 설계가 없으면 수정 속도가 오히려 느려진다.

## 핵심 개념
### 1) 테스트 계층 분리
단위/통합/부하 테스트는 목적이 다르다. 각 테스트의 책임을 명확히 해야 유지 비용이 줄고 신뢰도가 높아진다.

### 2) 실패 재현 가능성
Mock/Stub 전략의 본질은 '통과'보다 '실패를 재현하고 고정하는 능력'이다.

### 3) 운영 전 품질 확인
성능 한계와 장애 조건을 배포 전에 확인해야 운영 사고 확률을 줄일 수 있다.

## 프론트엔드에서 백엔드로 확장할 때의 관점
- 프론트 변경이 API 계약을 깨는지 자동 검증할 수 있다.
- 테스트를 기반으로 API 문서 신뢰도를 높일 수 있다.
- 배포 전 성능 리스크를 수치로 확인할 수 있다.

## 실무에서 자주 생기는 오해
- E2E 중심으로만 테스트한다.
- Mock 경계를 정의하지 않는다.
- 부하 테스트를 생략한다.

## 정리
Mock/Stub 전략는 품질을 감각이 아닌 지표로 관리하게 만드는 단원이다.
