# 8. 19주차 정리

## 주차 목표

- Docker 기반 배포 환경 표준화 개념 학습
- 컨테이너/이미지/Dockerfile/볼륨의 관계 이해
- 서버 환경 재현성을 확보하는 배포 흐름 설계

## 핵심 개념

### 1) Docker와 가상화

- Docker는 컨테이너 기반 가상화 플랫폼
- VM이 OS 단위 가상화에 가깝다면, Docker는 애플리케이션 단위 격리에 강점
- 여러 프로젝트를 한 호스트에서 충돌 없이 운영하기 유리

### 2) Docker가 필요한 이유

- 개발 환경 독립화: 버전/권한/의존성 충돌 완화
- 개발-배포 환경 일치: "내 PC에서는 됨" 문제 축소
- 복제 용이성: 같은 이미지로 동일 컨테이너 재생성

### 3) 핵심 구성요소

- Image: 컨테이너 실행 템플릿
- Dockerfile: 이미지 생성 명세서
- Layer: Dockerfile 단계별 캐시 단위
- Docker Hub: 이미지 레지스트리

### 4) 데이터 영속성

컨테이너 내부 데이터는 휘발되므로 외부 저장 전략 필수:

- Volume: Docker 관리 저장소
- Bind Mount: 호스트 디렉터리 직접 마운트

### 5) 운영 관점 원칙

- Docker는 배포/실행 표준화 도구로 활용
- 코드 자체는 Git 관리가 기본
- DB 데이터는 컨테이너 내부에만 의존하지 않도록 분리

## 과제 요구사항 정리

- 과제 서버 환경을 컨테이너로 즉시 실행할 수 있도록 `Dockerfile` 작성
- 컨테이너 내부 DB 테이블 생성 책임을 과도하게 넣지 않고, 데이터 계층 분리

## 실무 관점 체크포인트

- Dockerfile은 단순 설치 스크립트가 아니라 서비스 재현 계약서
- 이미지 경량화, 비밀정보 분리, 환경변수 주입, 헬스체크까지 고려하면 배포 품질이 크게 올라감
