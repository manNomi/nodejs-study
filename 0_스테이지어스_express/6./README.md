# 6. 17주차 정리

## 주차 목표

- 캐싱의 목적(속도/부하완화)과 부작용(메모리/휘발성) 이해
- Redis 자료구조를 서비스 시나리오에 맞게 선택
- 세션/로그인/검색어 같은 실전 기능을 Redis로 설계

## 핵심 개념

### 1) 캐싱 전략

- 캐시는 느린 저장소 접근을 줄여 응답속도를 높이는 전략
- DB I/O를 줄여 병목과 과부하를 완화
- 메모리 비용이 크고 휘발성이 있으므로 "무엇을 캐시할지" 선택이 중요

### 2) Redis 특성

- 메모리 기반 Key-Value/Collection 저장소
- 매우 빠른 읽기/쓰기 성능
- 영속성은 기본 보장되지 않아 별도 전략 필요(AOF/RDB)

### 3) Redis 자료구조 선택

- String: 단순 키-값
- Hash: 객체형 데이터
- List: 순서 있는 목록
- Set: 중복 제거
- Sorted Set: 점수 기반 정렬 + 중복 관리

### 4) 설계 시 주의점

- 한 컬렉션에 과도한 데이터 집중 금지
- 탐색 성능을 위해 키 설계와 분할 전략 필요
- 키 네이밍 예: `cart_{accountId}`, `search_recent_{accountId}`

### 5) 영속성 전략

- AOF: 명령 로그 기반 복구, 정밀하지만 파일 관리 필요
- RDB Snapshot: 시점 백업 방식, 운영 배치와 연계 필요

## 과제 요구사항 정리

- 로그인 통계:
  - 최근 1시간 로그인 회원 수
  - 전체 누적 로그인 수
  - 집계값을 RDB와 연동
- 검색 기능:
  - 최근 검색어 5개 유지
  - 중복 제거 + 최신순 재정렬
- Redis 세션 적용
- 중복 로그인 방지

## 피드백 반영 포인트

- `express-session` 저장소를 Redis로 확장 가능하도록 조사
- API 작명은 REST 원칙 기반으로 재정비

## 실무 관점 체크포인트

- Redis는 성능 도구이지 단독 진실 저장소가 아님
- 통계/세션/임시데이터는 Redis, 최종 정합 데이터는 RDB로 역할 분리
