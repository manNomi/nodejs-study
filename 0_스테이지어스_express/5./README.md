# 5. 16주차 정리

## 주차 목표

- 인증(Authentication)과 인가(Authorization) 흐름을 분리해서 설계
- Session 기반 인증에서 Token(JWT) 기반 인증으로 전환
- API 접근 제어를 "소유권/권한/로그인 상태" 기준으로 구체화

## 핵심 개념

### 1) Stateless vs Stateful

- Stateless: 요청 간 상태를 서버가 보관하지 않음
- Stateful: 서버가 사용자 상태(Session 등)를 유지
- HTTP 자체는 Stateless이므로, 로그인 상태 유지를 위해 별도 메커니즘 필요

### 2) 인증 방법 비교

- 요청마다 사용자 식별값 전달: 구현은 단순하지만 조작 위험 큼
- Cookie 직접 식별값 저장: 자동 전송은 편하지만 값 위변조 위험 존재
- Session-Cookie: 서버 세션 + 쿠키 세션ID
- Token(JWT 등): 서명 기반 무결성 검증, 서버 메모리 부담 감소

### 3) JWT 구조와 검증

- `Header`: 토큰 타입, 발급자, 만료 정보
- `Payload`: 사용자 식별/권한 정보
- `Signature`: 비밀키 기반 서명

검증 핵심은 "Payload가 조작되지 않았는가"이며, Signature 확인이 필수.

### 4) 인증만으로 충분하지 않은 이유

토큰 검증 성공 이후에도 반드시 확인해야 할 것:

- 로그인 여부
- 권한(role)
- 리소스 소유권(내 글인지)
- 수정 가능 상태 여부

예: 게시글 수정은 `WHERE post_id = ? AND account_id = ?` 같은 소유권 조건이 동반되어야 안전.

### 5) 보안 운영 포인트

- 토큰 저장은 LocalStorage보다 Cookie(`httpOnly`, `secure`) 우선 고려
- 탈취 피해 완화를 위해 access token 수명 단축 + refresh token 전략 적용
- 비밀키는 `.env`로 분리하고 버전관리 제외

## 과제 요구사항 정리

세션 기반 로직을 토큰 기반으로 전환하고 아래를 반영:

- 소유권 검증
- 수정 가능 여부 검증
- 계정 권한 검증
- 로그인 필수 API 접근 제어

## 실무 관점 체크포인트

- 인증은 "신원 확인", 인가는 "행동 허용"임을 분리해서 코드화
- 인증 미들웨어와 권한/소유권 미들웨어를 분리하면 재사용성과 테스트성이 올라감
