# 2. 13주차 정리

## 주차 목표

- API를 한 파일에 몰아넣지 않고 라우터/모듈 단위로 분해
- REST 네이밍 원칙과 예외처리 기준을 명문화
- 비동기 처리(Callback, Promise, async/await)와 `fetch` 흐름 이해

## 핵심 개념

### 1) 서버 구조 분리

- `server.js`에 모든 API를 작성하면 규모가 커질수록 리팩터링 비용 급증
- 기능별 라우터 파일 분리 + 라우터가 호출하는 모듈 분리로 유지보수성 확보
- 경로 깊이 설계 예시:
  - 상위: `/account`
  - 하위: `/account/login`, `/account/find-id`

### 2) 입력값 상태 구분

- `""`: 빈 문자열 입력
- `null`: 값 자체가 비어 있는 상태
- `undefined`: key 자체가 없거나 미할당 상태

이 셋을 동일하게 취급하면 검증 누락/오작동이 발생하므로 분리 처리 필요.

### 3) REST API 작명 원칙

- HTTP Method에 이미 동사가 있으므로 경로에 동사 중복 지양
- 경로는 명사 중심, 가능하면 단순하고 일관되게
- 과도한 복수형/복합어 남발 지양, 깊이 구조로 의미 분리

### 4) 예외처리 체크리스트

- null/빈 문자열/undefined 여부
- 길이 제한
- 자료형 검증
- 형식 검증(정규표현식)
- 리스트 길이 제한
- 첨부파일 확장자/크기 제한

### 5) API 명세서의 필수 요소

- 엔드포인트
- 설명
- Base URL
- Method
- Request body/query/path 스키마
- 성공/실패 응답 형식

명세서가 부실하면 프론트엔드가 API를 안정적으로 연동할 수 없음.

### 6) 비동기 처리와 fetch

- 비동기 함수는 완료를 기다리지 않고 다음 코드로 진행
- 오래 걸리는 작업(DB, 네트워크, 타이머, 이벤트)에서 필수
- 처리 방식 발전 순서:
  - Callback
  - Promise
  - async/await
- `fetch`는 Promise 기반 HTTP 통신 내장 API

## 과제 요구사항 정리

- REST API 최소 15개 설계
- Express 서버와 MariaDB 연결

## 실무 관점 체크포인트

- API는 기능 구현만이 아니라 "프론트가 쓰기 쉬운 형태"가 중요
- 비동기 코드에서 성공/실패 흐름을 명시하지 않으면 장애 분석이 어려움
